SECTION 1 — FAILURE EXPLANATION
--------------------------------------------------------------------------------

FAIL 1 — ADD 0xFFFFFFFF+1 [carry]
  Signal    : flag_ovf (V)
  Expected  : 0
  Got       : 1
  All other outputs (result=0x00000000, Z=1, N=0, C=1) are correct.

  In hardware terms, adding 0xFFFFFFFF (-1 in signed two's complement) and +1
  produces 0x00000000 with an unsigned carry-out of 1.  However, because the
  two operands have opposite signs (-1 and +1), a signed overflow is impossible
  by definition — no signed overflow can occur when the addends differ in sign.
  The overflow flag (V) must therefore be 0.

  The DUT asserts V=1 because BUG 1 replaces the dedicated signed-overflow
  signal (ovf_flag) with the unsigned carry-out (comb_carry) in the output
  register.  Since comb_carry=1 for this operation, flag_ovf is incorrectly
  driven high, falsely signalling a signed overflow that did not occur.  Any
  downstream signed-branch instruction consuming this flag would misbehave.

----------------------------------------

FAIL 2 — ADD 0x7FFFFFFF+1 [pos ovf]
  Signal    : flag_ovf (V)
  Expected  : 1
  Got       : 0
  All other outputs (result=0x80000000, Z=0, N=1, C=0) are correct.

  Adding the largest positive 32-bit signed integer (0x7FFFFFFF = +2147483647)
  and +1 wraps into the negative range (0x80000000 = -2147483648).  This is a
  textbook positive signed overflow: both inputs are positive, the result is
  negative.  The overflow flag (V) must be 1.

  The unsigned carry-out of this addition is 0 (no carry out of bit 32), so
  BUG 1's substitution of comb_carry for ovf_flag produces V=0.  The hardware
  silently misses a real signed overflow, which would cause silent data
  corruption in any signed arithmetic pipeline that relies on this flag to
  detect out-of-range results.

----------------------------------------

FAIL 3 — SUB 10-3=7
  Signal    : flag_ovf (V)
  Expected  : 0
  Got       : 1
  All other outputs (result=0x00000007, Z=0, N=0, C=1) are correct.

  Subtracting 3 from 10 in signed arithmetic produces +7, a value well within
  the representable range [-2147483648, +2147483647].  No signed overflow
  occurs.  The carry flag C=1 correctly indicates "no borrow" (A >= B in the
  unsigned sense, using the adder convention where SUB = A + ~B + 1).

  BUG 1 feeds the adder carry-out (comb_carry=1) directly into flag_ovf.
  Because there is no borrow (carry=1), the flag is spuriously set.  A
  processor relying on the overflow flag after this subtraction would
  incorrectly conclude the result overflowed a signed range.

FAIL 4 — SUB 5-5 [zero]
  Signal    : flag_ovf (V)
  Expected  : 0
  Got       : 1
  All other outputs (result=0x00000000, Z=1, N=0, C=1) are correct.

  Subtracting equal values produces zero with no borrow and no signed overflow.
  The carry-out is 1 (representing "no borrow" in the two's-complement
  subtraction convention).  BUG 1 once again routes this carry-out to flag_ovf,
  incorrectly asserting overflow for a completely benign subtraction of equal
  values.  A signed-compare operation checking A==B via subtraction would read
  a false overflow flag.

FAIL 5 — SRA 0xFFFFFFF8>>>1 [signext]
  Signal    : result, flag_neg (N)
  Expected  : result=0xFFFFFFFC, N=1
  Got       : result=0x7FFFFFFC, N=0

  Arithmetically right-shifting 0xFFFFFFF8 (-8 in signed 32-bit) by 1 position
  should produce 0xFFFFFFFC (-4).  The vacated MSB (bit 31) must be filled with
  the original sign bit (1) to preserve the sign — this is the defining
  difference between arithmetic shift right (SRA) and logical shift right (SRL).

  BUG 3 forces shift_fill=0 unconditionally, so the vacated MSB is filled with
  0 instead of 1.  The result becomes 0x7FFFFFFC (+2147483644), which is a large
  positive number — the exact opposite sign.  flag_neg is also wrong (0 instead
  of 1) as a direct consequence of the incorrect MSB.  In hardware, SRA is used
  to implement signed division by powers of two; this bug turns every such
  division of a negative number into a large positive result.

----------------------------------------

FAIL 6 — SRA 0x80000000>>>31 [allF]
  Signal    : result, flag_neg (N)
  Expected  : result=0xFFFFFFFF, N=1
  Got       : result=0x00000001, N=0

  Arithmetically right-shifting 0x80000000 (the most-negative 32-bit signed
  value, -2147483648) by 31 positions should propagate the sign bit (1) across
  all 32 bit positions, producing 0xFFFFFFFF (-1).  This is the maximum sign
  extension: 31 vacated positions all filled with 1.

  BUG 3 fills every vacated position with 0.  The 31-position zero-fill shift
  leaves only the original MSB (1) shifted all the way to bit position 0,
  yielding 0x00000001 (+1).  All 31 sign-fill bits are wrong.  flag_neg is
  incorrectly 0.  This completely inverts the sign of the result: -2^31 shifted
  right by 31 should give -1, but the buggy DUT returns +1.

BUG 1 — Overflow Flag (causes Fails 1, 2, 3, 4)
  RTL Section : Section 7 — Output registers (out_reg always_ff block)
  Logic type  : REGISTERED OUTPUT — the error is in an always_ff assignment,
                not in combinational logic.  The combinational overflow detector
                (ovf_flag, computed in Section 2) is correct; the fault is that
                the wrong signal is latched into the output register.
  Exact line  : flag_ovf <= is_arith ? comb_carry : 1'b0;

  comb_carry is add_out[32] — the unsigned carry-out of the 33-bit shared adder.
  ovf_flag is computed in Section 2 as:
      (~(operand_a[31] ^ b_inv[31])) & (operand_a[31] ^ add_out[31])
  This detects signed overflow: both inputs same sign, result opposite sign.
  These two signals are orthogonal conditions.  Using comb_carry in place of
  ovf_flag creates both false positives (carry=1, overflow=0: Fails 1,3,4) and
  false negatives (carry=0, overflow=1: Fail 2).

BUG 2 — DIV/MOD Guard (latent — 0 failures in this run)
  RTL Section : Section 5 — Result mux (result_mux always_comb block),
                ALU_DIV and ALU_MOD case arms
  Logic type  : COMBINATIONAL LOGIC — a guard condition inside always_comb.
  Exact lines : if (operand_b == 32'b0)   [in both ALU_DIV and ALU_MOD arms]

  The guard should check b_gated (the power-isolation-gated operand B) rather
  than operand_b (the raw module input).  For DIV/MOD operations needs_b=1, so
  b_gated == operand_b in all 38 current test vectors — the signals are
  identical, masking the bug entirely.  The divergence would surface if this
  ALU were embedded in a system where operand_b is non-zero but b_gated is
  forced to zero by a higher-level power management signal, or in a gate-level
  simulation with ICG cells active.  The bug is latent but architecturally
  incorrect.

BUG 3 — SRA Sign Extension (causes Fails 5, 6)
  RTL Section : Section 3 — Unified barrel shifter, shift_fill assignment
  Logic type  : COMBINATIONAL LOGIC — a continuous assign statement.
  Exact line  : assign shift_fill = 1'b0;

  The 5-level right-shift mux tree uses shift_fill as the bit replicated into
  all vacated MSB positions during right shifts.  For SRA, shift_fill must be
  operand_a[31] (the sign bit of the input).  For SRL and (indirectly) SLL,
  shift_fill must be 0.  The constant-zero assignment eliminates sign extension
  entirely: every SRA operation with a negative input (MSB=1) behaves
  identically to SRL, corrupting the result and the flag_neg output.
